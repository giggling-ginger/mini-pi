Sessions · JSONL on disk

Why a chat can
survive a restart

An agent’s memory is just the in-memory messages[] array. Session does something simple: write it to disk line by line, then on the next start read it back into the loop.

Source: src/session.ts + CLI flags --continue / --resume.

01 What happens without a session

Memory only

In the REPL, history is just an array. Process exits → array is gone → the model has no memory of the file context you just built (the code on disk still exists, of course).

With a session

After each turn, new messages are appended to a .jsonl file. Next time, --continue loads them → same history → the model picks up where you left off.

Session ≠ vector DB / long-term memory. It’s just “serialize chat messages.” Simple, diffable, human-readable — enough for learning a harness.

02 Where the files live

your-project/
├── src/ ...
└── .mini-pi/                 # gitignored
    └── sessions/
        ├── 2026-07-11T23-09-37_e497cd.jsonl
        └── 2026-07-12T10-02-11_a1b2c3.jsonl
  • Default dir: <cwd>/.mini-pi/sessions/ (lives with the project)
  • Filename: timestamp + short random string, used as the session id
  • --session-dir can change the location

03 What JSONL looks like

JSONL = JSON Lines: one JSON object per line, separated by newlines. Great for append-only writes — a crash is unlikely to corrupt the whole file. 

// Line 1 is always meta (session card)
{"type":"meta", "id":"2026-...", "title":"create hello.ts", ...}

// Every line after is a message (same roles as the API)
{"type":"message", "ts":"...", "message":{ "role":"user", "content":"..." }}
{"type":"message", "ts":"...", "message":{ "role":"assistant", "content":"...", "tool_calls":[...] }}
{"type":"message", "ts":"...", "message":{ "role":"tool", "tool_call_id":"...", "content":"..." }}

What gets stored

user / assistant (including tool_calls) / tool results. That’s the context the model needs on the next turn.

What doesn’t

The system prompt is not written to the file — it’s regenerated on each start from the current cwd. Cleaner when you change directories or prompt logic.

04 How it hooks into the agent loop

Session does not change the loop — it wraps around it:

1

Open / create session

New session writes one meta line; --continue loads JSONL → history[]

2

runAgent(prompt, history)

Inside: system + history + user → model → tools… returns the full non-system message list

3

syncFromHistory(newHistory)

Compare lengths; append only the new tail into the jsonl (not a full rewrite of every line)

4

Next turn / exit

In-memory history keeps going; if the process dies, disk already has the full record

Core mental model

// Startup
let session = Session.create(...)  // or Session.load(...)
let history = session.history       // [] or previous messages

// After each user turn
history = await runAgent(opts, userText, history)
session.syncFromHistory(history)   // append only new messages to disk
Why only append the tail? runAgent returns the full conversation; the session already knows it has N messages, so it only writes history.slice(N) as new lines. JSONL stays append-only — simple and reliable.

06 How commands map to the mechanism

CommandWhat happens on disk
Default start create a new file, write one meta line
--continue / -c find the latest jsonl by updatedAt, load all message lines
--resume list sessions; you pick one, then load
--session id resolve by id / partial id / path, then load
--list-sessions read each file’s meta + count message lines
--no-session no create, no read/write (memory only)
REPL /reset create another jsonl (old file is kept)
REPL /session print current id and path (does not change the file) 
# Start a new turn and save it
npm run dev -- -p "create hello.ts"

# Continue the chat
npm run dev -- -c -p "add a test"

# List existing sessions
npm run dev -- --list-sessions

07 Easy misconceptions

MisconceptionReality
Session remembers every project file Only chat messages; reading code still uses the read/bash tools
/reset deletes history files It just switches to a new file; old jsonl stays in the directory
Continue = one long-lived model request Each turn is a new API call — messages are just longer
meta.model forces the old model Currently just recorded; the live MODEL env wins
Same as Pi’s session tree Ours is a linear JSONL — no branch/fork (intentionally simplified)
In one line: Session = a disk mirror of messages[]. The agent loop is still the star; session only lets the loop’s input outlive the process.
