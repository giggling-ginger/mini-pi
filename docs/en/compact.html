Compaction · Context window

When the chat gets long
how we fold older turns

Model context is finite. Sessions grow turn by turn; tool output is especially expensive. Compaction = fold earlier messages into a summary, keep the recent turns verbatim, then continue the loop.

Source: src/compact.ts (called at the start of runAgent)

01 Why it matters

Without compaction

messages grow without bound → hit the model limit → requests fail, or cost explodes while the model “forgets” early goals and you still pay full price.

With compaction

Older turns become a summary; recent detail stays. The agent still knows “what we’re doing,” but token usage drops.

How this splits with Session / AGENTS.md: Session owns persistence; AGENTS.md is project instructions; Compaction keeps history from blowing the window. The three are orthogonal.

02 Mental model first

What the model always sees is a messages[] array. Compaction does not change the agent loop—it only shortens that array before the loop starts. 

// before (long)
[ u1, a1, t1, u2, a2, t2, … u20, a20, t20 ]

// after (short)
[ summaryUser, ackAssistant,  …recent verbatim turns… ]
     ↑ the big old blob becomes these two          ↑ detail kept
Analogy: writing a novel outline. Chapters 1–15 collapse into a “previously on…” blurb; the last 2 chapters stay in full—so the next chapter keeps the plot without pasting the whole book into the editor.

03 Source walkthrough

Files: src/compact.ts + src/agent.ts. Read in call order.

Entry: who calls it, and when?

Every runAgent(prompt, history), before assembling this turn’s messages: 

// agent.ts — start of runAgent
let hist = history.filter(m => m.role !== "system")

if (compact) {
  const result = await maybeCompact(client, model, hist, compactOptions)
  hist = result.history   // may now be “summary + recent”
  if (result.compacted) onEvent({ type: "compact", ... })
}

// then enter the normal loop
messages = [ system, ...hist, thisTurnUser ]

So compaction runs the moment “you just typed, the model has not started thinking yet.” This turn’s user message is not included yet—only history is compacted.

Step A — Should we compact? (estimate tokens)

// compact.ts
function estimateTokens(messages) {
  // sum chars from each message text + tool args
  return Math.ceil(totalChars / 4)
}

// inside maybeCompact:
if (!force && beforeTokens < threshold)  // default threshold=8000
  return { compacted: false, history: unchanged }  // skip, pass through

/4 is a rough estimate (English ≈ 1 token per 4 chars). We only need “are we roughly near the limit?”—not billing-grade accuracy.

Step B — Where to cut? splitOldRecent

Goal: keep about keepRecent (default 3000) tokens of recent at the tail; everything before is old.

// scan backward from the end, accumulate tokens until budget is used
for (i = lastMessage; i >= 0; i--) {
  if (addingThisWouldExceed keepRecent && recent already has items)
    break
  tokenBudget += thisMessage
  idx = i   // recent starts at idx
}

// important: slide idx right to the nearest user so we don’t cut mid-tool
while (messages[idx].role !== "user") idx++

Example (arrow = cut point):

u1 a1 t1 | u2 a2 t2 u3 a3
  old    ↑      recent
         must land on a user

// bad cut (will break):
u1 a1 | t1 u2 …     // t1 needs preceding a1.tool_calls; API rejects

Step C — Turn old into a short summary

Two paths:

llmSummary (default)

Flatten old into a plain-text transcript, then call chat.completions once more (no tools), asking the model for a bullet summary: goals, files touched, commands, errors, current state.

localSummary (tests / offline)

No API: one truncated bullet per user/assistant/tool message. Crude but free—handy for unit tests.

Step D — Rebuild history

// end of maybeCompact
folded = [
  {
    role: "user",
    content: "[Conversation summary — earlier turns compacted]\n" + summary
  },
  {
    role: "assistant",
    content: "Understood. I have the summarized context…"
  },
  ...recent   // recent turns, byte-for-byte
]
return { history: folded, compacted: true, beforeTokens, afterTokens, ... }

Why a user+assistant pair instead of a single system message? Because we never persist system in history (system is rebuilt each turn); a user/assistant pair sits naturally in the session JSONL and reloads cleanly when you resume.

Step E — Write back to the session

After compaction the array prefix changes (not a simple append), so session.syncFromHistory does a full-file rewrite of the JSONL, not just append lines at the end.

04 A full numeric example

Assume threshold=100, keepRecent=40 (scaled down so the numbers are easy).

// history before (≈150 tok)
0 user:       create app
1 assistant:  write package.json  (tool write)
2 tool:       Wrote …
3 user:       add routes
4 assistant:  edited router.ts
5 tool:       …
… (lots in the middle)
18 user:      fix tests
19 assistant: run vitest
20 tool:      3 failed …

// split: accumulate ~40 tok from the end → land on a user, e.g. index 16
old    = [0 .. 15]     // will be folded
recent = [16 .. 20]    // kept verbatim

// llm summarizes old → a few bullets
summary = "- built app and routes\n- edited router.ts\n- tests still failing…"

// history after (≈55 tok)
0 user:      [Conversation summary…] + summary
1 assistant: Understood…
2 user:      fix tests          // was 18
3 assistant: run vitest
4 tool:      3 failed …

When you later say “keep fixing,” the model sees: “tests are failing” from the summary + the recent vitest output verbatim. It loses some unrelated mid-chat chatter, but keeps the error log that just happened.

05 Algorithm flow (by function)

runAgent()
  │
  ├─ estimateTokens(history)          // A estimate size
  │     └─ < threshold? → no compact, return original history
  │
  ├─ splitOldRecent(history, keep)    // B split
  │     ├─ accumulate tokens from the tail
  │     └─ align to a user boundary
  │
  ├─ llmSummary(old) or localSummary  // C lossy compress
  │
  ├─ folded = [summaryUser, ack, ...recent]  // D rebuild
  │
  └─ messages = [system, ...folded, newUser]
        → normal tool loop…
Key: this is lossy compression. Unlike zip, you cannot restore the original text. You trade context space for possible loss of older detail. So don’t set keepRecent too small—the last few turns (especially tool output) usually matter most.

08 Common pitfalls

PitfallNotes
Summary drops key detail keepRecent too small loses recent code; raise keep
Token estimate is rough chars/4 is approximate; fine as a trigger, not for billing
Session file After compact the history prefix changes → mini-pi rewrites the whole JSONL
Re-compaction The summary stays; later growth can compact again; old summaries may fold into a newer one
One-liner: Compaction = lossy compression of conversation history. Trade space (context window) for precision (detail)—unlike zip, you cannot decompress back to the original.
