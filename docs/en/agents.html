AGENTS.md · Project instructions

How project rules enter
the model’s system prompt

mini-pi’s system prompt is intentionally short. Project conventions (package manager, test commands, directory layout…) are not hard-coded—they are read from AGENTS.md in the repo and appended to system.

Source: src/context-files.ts · src/system.ts

01 Why you need it

Without AGENTS.md

The model only knows “you are a coding agent + four tools.” It does not know whether you use pnpm or npm, how to run tests, or which directories to leave alone.

With AGENTS.md

Loaded into the system prompt at startup. Every API round-trip carries these rules (system is not stored in the session file, but the latest on-disk content is reloaded each time).

Division of labor with Session: Session remembers “what we talked about”; AGENTS.md remembers “the rules of this repo.” The former is conversation history; the latter is the project manual.

02 How files are “discovered”

Starting from cwd, walk up to the git root (the level that has .git), looking along the way for:

  • AGENTS.md (preferred)
  • else CLAUDE.md
  • else AGENT.md

Only one file per directory; files closer to cwd come later (more specific).

repo/                          # git root = stop walking up
├── AGENTS.md                  # ① collected first (global conventions)
├── package.json
└── packages/app/              # you run mini-pi here
    ├── AGENTS.md              # ② collected later (package-level)
    └── src/...
Budget: all context files combined are capped at about 32KB of characters; overflow is truncated and marked [truncated], so one giant markdown file cannot blow up the context.

03 How it is injected into the prompt

Each round of messages sent to the model is roughly this “sandwich”:

system · base (hard-coded, very short)
You are a coding agent… four tools: read, write, edit, bash…
system · project instructions (AGENTS.md appended)
# Project instructions ### AGENTS.md Use pnpm. Run tests with vitest… ### packages/app/AGENTS.md Prefer colocated tests…
history (from session, no system)
user / assistant / tool / …
this turn’s user
add a failing test for parseArgs
// system.ts (simplified)
const files = loadContextFiles({ cwd })
const prompt =
  buildBaseSystemPrompt(cwd) +
  formatContextBlock(files)   // empty string if no files

messages = [
  { role: "system", content: prompt },
  ...history,
  { role: "user", content: userMessage },
]

So: edit AGENTS.md and start a new turn (or a new session), and the model sees the new rules immediately— no need to clear the session (system is rebuilt every time).

05 What belongs in AGENTS.md

Good fits

Package manager, test/build commands, directory conventions, hard don’ts, PR/commit style, and tool preferences specific to this repo.

Don’t bloat it

Full design docs, giant API lists, or temporary TODOs that will go stale—they burn context and distract the model. Keep it short and actionable.

CommandEffect

--no-context-files / -nc Skip loading for this run (handy for A/B checks)

npm run dev -- -p "run typecheck"
# stderr may show: context 1 file(s): AGENTS.md

npm run dev -- -nc -p "same task, but without AGENTS.md"

07 Common misconceptions

MythReality
AGENTS.md is stored in the session JSONL It is not. System is rebuilt every time; file edits take effect immediately
Same as skills / MCP Just a text manual—it does not register new tools
The model will obey 100% Still soft instructions; clear and short works better
Subdir CLAUDE.md and AGENTS.md are merged Only the highest-priority name is taken per directory
In one line: AGENTS.md = a version-controlled project system appendix. The harness finds it and stitches it into the prompt; how well you write it decides whether the agent feels like “one of your team.”
