Streaming · How it works

Why tokens can
pop out one by one

Streaming does not change the shape of the agent loop—it is still “model → tools → model.” It only changes how the assistant message reaches you: wait for the full package, or push as it is generated.

Source: streamCompletion in src/agent.ts.

01 Wait for the full reply vs stream

Non-streaming stream: false

One HTTP request: the server finishes thinking the entire reply, then returns JSON. The terminal is quiet for a few seconds, then a whole block appears. Simple to implement; feels like a hitch.

Streaming stream: true (default)

In the same request, the server pushes small chunks as it generates. You see text appear immediately; tool arguments are assembled piece by piece as well.

Core mental model: the model only “thinks” a little at a time (tokens). Non-streaming means the server buffers tokens into a full reply before sending; streaming means it gives you each bit as it is ready. Agent logic lives on the client—you still must wait for complete tool_call arguments before running a tool.

02 What it looks like on the wire

Streaming for OpenAI-compatible APIs is usually SSE (Server-Sent Events): response header Content-Type: text/event-stream, body lines of data: {...}, ending with data: [DONE].

Each data line is a small JSON chunk; the core is choices[0].delta:

{
  "id": "chatcmpl-...",
  "object": "chat.completion.chunk",
  "choices": [{
    "index": 0,
    "delta": {
      // text delta (may be only a few characters)
      "content": "Hel"
      // or tool deltas (see next section)
    },
    "finish_reason": null
  }]
}

The SDK (we use the openai package) turns SSE into an async iterator: for await (const chunk of stream). You do not need to parse SSE by hand, but you should know that every chunk is incomplete.

Timeline sketch (plain text, no tools)

client
POST /v1/chat/completions { stream: true, messages, tools }
server
delta.content = "I'll"
server
delta.content = " create"
server
delta.content = " hello.ts"

finish_reason = "stop" · then [DONE]

assemble full content → no tool_calls → end this turn

03 The hard part: tool_call is also “fragments”

If the model decides to call write, the argument JSON is not delivered in one chunk. It is split into many small pieces—same as text, pushed as it is generated.

Two kinds of “index” to keep straight

Chunk order (time order)

Packet 1, packet 2, packet 3 on the wire… that is only arrival order. There is no field called chunk_id in the API; “chunk 1/2/3” in comments below is just for reading.

index (tool-call slot)

index: 0 = the first tool this assistant turn will call. If two tools run in parallel, you get index: 0 and index: 1. With a single tool, every fragment is index 0—that is normal!

Analogy: you are assembling one letter (one tool call). The post office delivers 4 slips in 4 trips; each says “this is a fragment of letter 0,” with content like {"path":, "hello.ts",… You glue them in order into complete JSON. It is not “4 different letters,” so index stays 0 the whole time.

With only 1 tool: index stays 0

Each block below is a real delta from the server; gray comments show client state after assembly.

// ── packet 1: create slot 0, include tool name ──
delta.tool_calls = [{
  index: 0,
  id: "call_abc",
  function: { name: "write", arguments: "" }
}]
// after assemble → tools[0] = { name: "write", arguments: "" }

// ── packet 2: append argument fragments to the same slot ──
delta.tool_calls = [{
  index: 0,
  function: { arguments: '{"path":' }
}]
// after assemble → tools[0].arguments = '{"path":'

// ── packet 3 ──
delta.tool_calls = [{
  index: 0,
  function: { arguments: '"hello.ts",' }
}]
// after assemble → tools[0].arguments = '{"path":"hello.ts",'

// ── packet 4 ──
delta.tool_calls = [{
  index: 0,
  function: { arguments: '"content":"hi"}' }
}]
// after assemble → tools[0].arguments = '{"path":"hello.ts","content":"hi"}'  ✅ complete

// stream ends finish_reason = "tool_calls"
// → only now executeTool("write", full JSON)

With 2 tools in parallel: then index 1 appears

// same turn: model says read then bash (two parallel tool_calls)

// some packet: fill slot 0
{ index: 0, function: { name: "read",  arguments: '{"path":"a.ts"}' } }

// some packet: fill slot 1  ← this is the second tool
{ index: 1, function: { name: "bash",  arguments: '{"command":"ls"}' } }

// client must Map by index and assemble separately — not one big string
Never call executeTool before arguments are fully assembled— half JSON will fail to parse, or corrupt a file. mini-pi accumulates with Map<index, {id,name,arguments}>, and only after the stream ends turns that into complete tool_calls, then executes.

So one turn under streaming is really:

  • Receive stream: print content deltas; assemble arguments per index
  • Stream done: you have a full assistant message (same shape as a non-streaming response)
  • Run tools (if any) → push results into messages → open the next stream turn

05 How mini-pi implements it

Turn on stream

const stream = await client.chat.completions.create({
  model,
  messages,
  tools: toolDefinitions,
  tool_choice: "auto",
  stream: true,  // ← just this switch
});

Accumulate content + tool_calls

let content = "";
const tools = new Map<number, { id: string; name: string; arguments: string }>();

for await (const chunk of stream) {
  const delta = chunk.choices[0]?.delta;
  if (delta?.content) {
    content += delta.content;
    onEvent?.({ type: "text_delta", text: delta.content });  // print to terminal immediately
  }
  for (const tc of delta?.tool_calls ?? []) {
    const acc = tools.get(tc.index) ?? { id: "", name: "", arguments: "" };
    if (tc.id) acc.id = tc.id;
    if (tc.function?.name) acc.name += tc.function.name;
    if (tc.function?.arguments) acc.arguments += tc.function.arguments;
    tools.set(tc.index, acc);
  }
}
// only after the stream ends are tools ready as executable tool_calls

How the CLI “types”

On text_delta, write with process.stdout.write(delta)— do not println every time, or each token becomes its own line and looks terrible. Add a newline before tools start. 

# streaming on by default
npm run dev -- -p "write a hello.ts"

# compare: full package return
npm run dev -- --no-stream -p "write a hello.ts"

06 Common pitfalls

PitfallNotes
Execute on half JSON Wait until the stream ends (or finish_reason) before running tools
Drop index With parallel tools, accumulate arguments separately per index
name can also fragment Some backends split the function name; += is safer
Relation to history What you write into messages is still a complete assistant message, not a pile of chunks
Free models / proxies Some endpoints are flaky with streaming; use --no-stream to compare and debug
In one line: Streaming = transport optimization; Agent loop = control flow. Do not mash them together—first assemble the same message shape as non-streaming, then run the usual tool loop.
