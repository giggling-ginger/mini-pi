Sessions · JSONL on disk

对话为什么能
关掉再续上

Agent 的记忆就是内存里那串 messages[]。 Session 做的事很朴素：把它一行行写到磁盘；下次启动再读回来塞进 loop。

对应源码：src/session.ts + CLI 里的 --continue / --resume。

01 没有 session 时会发生什么

只有内存

REPL 里 history 是个数组。进程退出 → 数组没了 → 模型完全不记得你刚让它写的文件上下文（当然磁盘上的代码还在）。

有 session

每一轮结束后把新消息 append 进 .jsonl。 下次 --continue 加载 → 同一串 history → 模型接着聊。

Session ≠ 向量数据库 / 长期记忆。 它就是「把 chat messages 序列化」。简单、可 diff、人眼能读——学习 harness 刚好够用。

02 文件放哪

你的项目/
├── src/ ...
└── .mini-pi/                 # gitignore 掉了
    └── sessions/
        ├── 2026-07-11T23-09-37_e497cd.jsonl
        └── 2026-07-12T10-02-11_a1b2c3.jsonl
  • 默认目录：<cwd>/.mini-pi/sessions/（跟项目走）
  • 文件名：时间戳 + 短随机串，当作 session id
  • --session-dir 可改存放位置

03 JSONL 长什么样

JSONL = JSON Lines：一行一个 JSON 对象，用换行分隔。 适合「只往末尾追加」，崩溃了也不容易整文件写坏。 

// 第 1 行永远是 meta（会话名片）
{"type":"meta", "id":"2026-...", "title":"create hello.ts", ...}

// 之后每一行是一条 message（和 API 里的 role 一致）
{"type":"message", "ts":"...", "message":{ "role":"user", "content":"..." }}
{"type":"message", "ts":"...", "message":{ "role":"assistant", "content":"...", "tool_calls":[...] }}
{"type":"message", "ts":"...", "message":{ "role":"tool", "tool_call_id":"...", "content":"..." }}

会存什么

user / assistant（含 tool_calls）/ tool 结果。这些是模型下一轮要看见的上下文。

不存什么

system prompt 不进文件——每次启动按当前 cwd 重新生成。 换目录、改 prompt 逻辑时更干净。

04 和 Agent Loop 怎么接

Session 不改变 loop，只包在外面：

1

打开 / 创建 session

新会话写一行 meta；--continue 则 load JSONL → history[]

2

runAgent(prompt, history)

内部：system + history + user → 模型 → 工具… 返回完整非 system 消息列表

3

syncFromHistory(newHistory)

对比旧长度，把新增的尾巴 append 进 jsonl（不是整文件重写每条）

4

下一轮 / 退出

内存里 history 继续用；进程挂了也没关系——磁盘上已有完整记录

核心代码心智

// 启动
let session = Session.create(...)  // 或 Session.load(...)
let history = session.history       // [] 或上次的 messages

// 每一轮用户输入之后
history = await runAgent(opts, userText, history)
session.syncFromHistory(history)   // 只把新消息 append 到磁盘
为什么只 append 尾巴？ runAgent 返回的是「整段对话」；session 记得自己已经有 N 条， 只把 history.slice(N) 写成新行。这样 JSONL 保持追加写，简单可靠。

06 命令怎么映射到原理

命令磁盘上发生什么
默认启动 create 新文件，写一行 meta
--continue / -c 按 updatedAt 找最新 jsonl，load 全部 message 行
--resume 列出 sessions，你选一个再 load
--session id 按 id / 部分 id / 路径 resolve 后 load
--list-sessions 读每个文件的 meta + 数 message 行
--no-session 不创建、不读写文件（纯内存）
REPL /reset 新建另一个 jsonl（旧文件保留）
REPL /session 打印当前 id 与路径（不改文件） 
# 新开一轮并保存
npm run dev -- -p "create hello.ts"

# 接着聊
npm run dev -- -c -p "再加个 test"

# 看看有哪些会话
npm run dev -- --list-sessions

07 容易误解的点

误解实际
Session 会记住所有项目文件 只记对话消息；读代码仍靠 read/bash 工具
/reset 会删历史文件 只是换新文件；旧 jsonl 还在目录里
续聊 = 同一个模型请求一直开着 每次都是新的 API 调用，只是 messages 更长
meta 里的 model 会强制用旧模型 目前只作记录；真正用的是你当前 env 的 MODEL
和 Pi 的 session 树一样 我们是线性 JSONL，没有 branch/fork（有意简化）
一句话：Session = messages[] 的磁盘镜像。 Agent loop 仍然是主角；session 只是让 loop 的输入能活过进程生命周期。
