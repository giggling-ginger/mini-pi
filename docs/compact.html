Compaction · Context window

对话太长时
怎么把旧轮次 折起来

模型上下文有限。Session 会越聊越长，tool 输出尤其占地方。 Compaction = 把较早的消息收成一段摘要，留下最近几轮原文，再继续 loop。

源码：src/compact.ts（在 runAgent 开头调用）

01 为什么需要

不压

messages 无限增长 → 触及模型上限 → 请求失败，或费用暴涨、模型「忘了」早期目标却仍付全价。

压

旧轮次变成摘要；近期细节保留。Agent 仍知道「我们在做什么」，但 token 用量掉下来。

和 Session / AGENTS.md 的分工： Session 负责持久化；AGENTS.md 是项目说明书；Compaction 负责历史别撑爆窗口。 三者正交。

02 mini-pi 怎么做

  1. 估 token — 粗算 chars / 4（够用，不必接 tokenizer）
  2. 过阈值？ — 默认约 8000；可用 --compact-threshold 调
  3. 切开 — 尾部保留 ~3000 token 的「recent」，尽量从某条 user 切开（避免拆开 tool 链）
  4. 摘要 old — 默认再调一次模型（无 tools）；失败/测试可用本地抽取摘要
  5. 替换 — [summary user] + [ack assistant] + recent

old（将被折叠）

更早的 user / assistant / tool… 可能很长

summary 对

// runAgent 开头（简化）
if (compact) {
  const result = await maybeCompact(client, model, history, opts)
  if (result.compacted) {
    history = result.history

}
messages = [ system, ...history, userMessage ]
为什么从 user 切开？ Assistant 的 tool_calls 必须和后面的 role: tool 成对。 若 recent 从半截 tool 结果开始，API 会报协议错误。

04 命令

命令作用
（默认） 每轮 runAgent 前检查是否需要 compact
--no-compact 关闭
--compact-threshold N 触发阈值（估 token）
--compact-keep N 保留的 recent 体量
/tokens 看当前 history 估多少 token
/compact 立刻强制压一轮并写回 session

05 常见坑

坑说明
摘要丢关键细节 keepRecent 太小会丢近期代码；可调大 keep
估 token 不准 chars/4 是粗算；够做触发器用，不是计费精度
session 文件 压缩后 history 前缀变了 → mini-pi 整文件 rewrite JSONL
二次压缩 摘要还在，后面又变长会再压；旧摘要可能再被折进更新的摘要
一句话：Compaction = 有损压缩对话历史。 换空间（上下文窗口）与精度（细节）——和 zip 不同，解压不回原文。
