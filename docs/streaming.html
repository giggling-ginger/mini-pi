Streaming · How it works

Token 为什么能
一个字一个字 蹦出来

Streaming 不改变 agent loop 的结构——还是「模型 → 工具 → 模型」。 它只改变助手那条消息是怎么到达你这边的：整包等完，还是边生成边推。

对应源码：src/agent.ts 里的 streamCompletion。

01 等整包 vs 流式

非流式 stream: false

一次 HTTP 请求，服务器把整句想完再返回 JSON。 终端安静几秒，然后突然整段出现。实现简单，体感像「卡一下」。

流式 stream: true（默认）

同一次请求里，服务器边生成边推送小块。 你立刻看到字往外蹦；工具参数也是一点点拼出来的。

关键心智模型：模型一次只「想」一点点（token）。 非流式是服务端把 token 攒成完整回复再给你；流式是每攒一点就给你一点。 Agent 逻辑在客户端——你仍然要等完整的 tool_call 参数才能执行工具。

02 线路上长什么样

OpenAI 兼容 API 的 streaming 通常是 SSE（Server-Sent Events）： 响应头 Content-Type: text/event-stream， 正文一行行 data: {...}，最后 data: [DONE]。

每个 data 里是一个小 JSON chunk，核心在 choices[0].delta：

{
  "id": "chatcmpl-...",
  "object": "chat.completion.chunk",
  "choices": [{
    "index": 0,
    "delta": {
      // 文本增量（可能只有几个字符）
      "content": "Hel"
      // 或工具增量（见下一节）
    },
    "finish_reason": null
  }]
}

SDK（我们用的 openai 包）把 SSE 解析成异步迭代器： for await (const chunk of stream)。 你不需要手写 SSE 解析，但要知道每个 chunk 都是不完整的。

时间线示意（纯文本，无工具）

client
POST /v1/chat/completions { stream: true, messages, tools }
server
delta.content = "I'll"
server
delta.content = " create"
server
delta.content = " hello.ts"
server
finish_reason = "stop" · 然后 [DONE]
client
拼出完整 content → 无 tool_calls → 结束本轮

03 难点：tool_call 也是「碎片」

若模型决定调用 write，参数 JSON 不会一个 chunk 给全。 而是拆成很多小片段——和文字一样，也是边生成边推。

先分清两个「编号」

chunk 序号（时间顺序）

网络上第 1 包、第 2 包、第 3 包……只是到达的先后。 API 里没有叫 chunk_id 的字段；下面注释里的 “chunk 1/2/3” 只是方便阅读。

index（工具调用槽位）

index: 0 = 这一轮 assistant 要调的第 1 个工具。 若一次并行调两个工具，会有 index: 0 和 index: 1。 只调一个时，所有碎片都是 index 0——正常！

类比：你在拼一封信（一个 tool call）。邮差分 4 次送来 4 张纸条， 每张都写着「这是第 0 封信的碎片」，内容分别是 {"path":、 "hello.ts",……你要按顺序粘成完整 JSON。 不是「4 封不同的信」，所以 index 一直是 0。

只有 1 个工具时：index 一直是 0

下面每一包都是服务器推来的真实 delta；灰色注释是客户端拼完后的状态。

// ── 第 1 包：新建槽位 0，带上工具名 ──
delta.tool_calls = [{
  index: 0,
  id: "call_abc",
  function: { name: "write", arguments: "" }
}]
// 拼完后 → tools[0] = { name: "write", arguments: "" }

// ── 第 2 包：往同一个槽位追加 arguments 碎片 ──
delta.tool_calls = [{
  index: 0,
  function: { arguments: '{"path":' }
}]
// 拼完后 → tools[0].arguments = '{"path":'

// ── 第 3 包 ──
delta.tool_calls = [{
  index: 0,
  function: { arguments: '"hello.ts",' }
}]
// 拼完后 → tools[0].arguments = '{"path":"hello.ts",'

// ── 第 4 包 ──
delta.tool_calls = [{
  index: 0,
  function: { arguments: '"content":"hi"}' }
}]
// 拼完后 → tools[0].arguments = '{"path":"hello.ts","content":"hi"}'  ✅ 完整

// 流结束 finish_reason = "tool_calls"
// → 现在才能 executeTool("write", 完整 JSON)

若一次并行调 2 个工具：才会出现 index 1

// 同一轮里模型说：先 read 再 bash（并行两个 tool_calls）

// 某包：往槽位 0 填
{ index: 0, function: { name: "read",  arguments: '{"path":"a.ts"}' } }

// 某包：往槽位 1 填  ← 这才是第二个工具
{ index: 1, function: { name: "bash",  arguments: '{"command":"ls"}' } }

// 客户端必须用 Map 按 index 分开拼，不能全糊在一个字符串上
绝对不能在参数还没拼完时就 executeTool—— 半截 JSON 会 parse 失败，或者写坏文件。 mini-pi 用 Map<index, {id,name,arguments}> 累加， 流结束后再变成完整的 tool_calls，然后才执行。

所以 streaming 下的一轮其实是：

  • 收流：content 增量打印；每个 index 各自拼 arguments
  • 收完：得到完整 assistant message（和一次非流式返回的形状一样）
  • 执行工具（若有）→ 结果塞回 messages → 再开下一轮 stream

模拟一次「创建 hello.ts」：文字先流出来，再组装 write 的参数，最后执行工具。

05 mini-pi 里怎么写

打开 stream

const stream = await client.chat.completions.create({
  model,
  messages,
  tools: toolDefinitions,
  tool_choice: "auto",
  stream: true,  // ← 就这一个开关
});

累加 content + tool_calls

let content = "";
const tools = new Map<number, { id: string; name: string; arguments: string }>();

for await (const chunk of stream) {
  const delta = chunk.choices[0]?.delta;
  if (delta?.content) {
    content += delta.content;
    onEvent?.({ type: "text_delta", text: delta.content });  // 立刻打到终端
  }
  for (const tc of delta?.tool_calls ?? []) {
    const acc = tools.get(tc.index) ?? { id: "", name: "", arguments: "" };
    if (tc.id) acc.id = tc.id;
    if (tc.function?.name) acc.name += tc.function.name;
    if (tc.function?.arguments) acc.arguments += tc.function.arguments;
    tools.set(tc.index, acc);
  }
}
// 流结束后 tools 才变成可执行的 tool_calls

CLI 怎么「打字」

text_delta 事件里直接 process.stdout.write(delta)， 不要每次 println——否则一个 token 一行，会很难看。 工具开始前再补换行。 

# 默认开流式
npm run dev -- -p "写个 hello.ts"

# 对比：整包返回
npm run dev -- --no-stream -p "写个 hello.ts"

06 常见坑

坑说明
半截 JSON 就执行 必须等流结束（或 finish_reason）再 run tool
丢掉 index 多工具并行时，按 index 分别累加 arguments
name 也可能分片 少数后端会把函数名拆开；用 += 更稳
和历史消息的关系 写入 messages 的仍是完整 assistant 消息，不是一堆 chunk
免费模型 / 代理 有的 endpoint 流式不稳；可用 --no-stream 对照排查
一句话：Streaming = 传输层优化；Agent loop = 控制流。 别把两者揉成一锅——先拼出和「非流式」一样的 message，再走原来的工具循环。
